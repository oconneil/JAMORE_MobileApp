ยิง GetUserCompany ต่อจาก Login

เป้าหมาย: สร้าง sequential API flow ที่ request ที่สองได้รับ token จาก request แรก โดยยังไม่เดาโครงสร้าง response

1. แยกหน้าที่ก่อน

AuthRepository รู้วิธี Login และเก็บ AuthSession

CompanyRepository รู้ endpoint ของ company

AppState เป็นผู้เรียง workflow: login → company → dashboard

ApiClient ทำ HTTP และแนบ Bearer token

2. สร้าง CompanyRepository รอบแรก

คืนค่าเป็น Object? ชั่วคราวเพื่อดู JSON จริงก่อน เทียบได้กับ unknown ใน TypeScript

abstract interface class CompanyGateway {
  Future<Object?> getUserCompany(String userName);
}

class CompanyRepository implements CompanyGateway {
  CompanyRepository(this._apiClient);
  final ApiClient _apiClient;

  @override
  Future<Object?> getUserCompany(String userName) {
    final encoded = Uri.encodeComponent(userName.trim());
    return _apiClient.get('Company/GetUserCompany/$encoded');
  }
}

3. ให้ AppState เรียงงาน

final session = await _authGateway.login(...);
final companyResponse = await _companyGateway.getUserCompany(
  session.userName,
);

// วาง breakpoint ตรงนี้ แล้ว inspect companyResponse
userCompanyResponse = companyResponse;

data = data.copyWith(sessionActive: true);
location = '/dashboard';

แต่ละ await คือจุดพักของ function นี้ งานสองชิ้นจึงเกิดตามลำดับ ไม่ได้ยิงพร้อมกัน

4. ทำไม token ถูกแนบได้

ก่อน AuthRepository.login() return มันตั้ง _session = session แล้ว เมื่อ company request วิ่งถึง ApiClient._headers() callback ใน main.dart จึงอ่าน token ล่าสุดและสร้าง Authorization: Bearer ...

กับดัก: ถ้ายิง company ภายใน AuthRepository ก่อนตั้ง _session request ที่สองจะไม่มี Bearer token

5. แบบฝึกลงมือ

  1. สร้าง gateway/repository และ register dependency ใน main.dart
  2. เพิ่ม dependency และ field ชั่วคราวใน AppState
  3. วาง breakpoint หลัง company call และที่ ApiClient._headers()
  4. ทำนาย token/response ก่อนรัน แล้วจด shape ของ response จริง
  5. ยังไม่สร้าง Company model จนกว่าจะเห็น JSON จริง

เกณฑ์ผ่านบทนี้

Network inspector เห็น Login ก่อน GetUserCompany, request ที่สองมี Bearer token, debugger เห็น response และ route ยังไม่เปลี่ยนก่อนทั้งสอง request สำเร็จ
